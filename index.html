Don't fork

The reason a separate mobile website is dangerous is that, in general, you want to avoid creating multiple versions of your website. It's called forking, and it's a forking nightmare from a maintenance POV. If you fork your website into separate mobile and desktop versions, then you're stuck updating both every time there's a change.

Don't create a silo team

Avoid creating a separate project for every device and save resources

The Web as a service:

If we consider:

  • "The Web" as the "Web Service",
  • "Websites" as the "Interface" and
  • "Device" as the "Environment".

The Web is unique, but:

  • the "Websites/Interface" will be just one of many interfaces we will use to interact with the "The Web/Web Service"

and..

  • the "Device/Environment" will be just one of many interfaces we will use to interact with the "Websites/Interface"

So in order to serve the same

  • "Websites/Interface" to different "Device/Environment" we need a technology to be smart enough to adapt to the new environments

Responsive Web Design is just the beginning:

  1. Responsive Web Design
    • Flexible layout
    • Flexible media
    • @media queries
  2. Mobile First Development
    • Instead of coding a large screen web experience first and then adding additional code on top, in order to work on smaller screens, we start from Mobile.
  3. Progressive Enhancement Development
    • Create the light core and progressively enhance the experience from there
  4. Usability Development
    • Mobile Optimisation
    • Touch Optimisation
    • Performance Optimisation
